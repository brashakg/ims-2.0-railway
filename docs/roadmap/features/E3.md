## Current behavior (file:line)

Counts `stock_units` where `status` is one of `["AVAILABLE", "available", "IN_STOCK", "in_stock"]` or status is absent/null. QUARANTINE, UNDER_AUDIT, and blind-count-flagged units have no status value that excludes them, so they are silently counted as on-hand sellable stock.

All query `status in ["AVAILABLE", "RESERVED"]` — same gap; QUARANTINE/UNDER_AUDIT would appear as on-hand if those statuses existed.

A separate independent collection (`serial_numbers`) with its own CRUD (`GET/POST/PATCH /inventory/serials`), a `_SERIAL_STATUSES` set `{"IN_STOCK","SOLD","WARRANTY_CLAIM","DAMAGED","LOST_STOLEN"}`, and a warranty-status derivation. ## Intended behavior (full intent)

Every physical state change to every stocked unit — mint, serial-bind, quarantine, audit-flag, transfer, scan-advance, reserve, sell, void — is recorded as one immutable row in `item_events`. These rows are:

- **Monotonically sequenced** per a shared `item_event_seq` counter (via the existing `barcode.allocate_sequence` pattern).
- **NOT hash-chained through `audit_chain.append_audit_entry`** (see CORRECTIONS P0-2 below — the global hash-chain serialises the POS sell-path and is not verified per-unit). Instead, each row carries its own `event_seq` and is append-only by policy (no update/delete route).
- **Append-only, single source of truth** for unit history; `stock_units.status` is a projection that E3 keeps in sync via a single-document `find_one_and_update` CAS per event.

Three new statuses are introduced — `QUARANTINE`, `UNDER_AUDIT`, and `BLIND_COUNT` — and every on-hand rollup in `_on_hand_by_product`, `find_available`, and all count/CL/sell-through aggregations must exclude them. POS `find_available` already returns only AVAILABLE; E3 makes the exclusion explicit and adds the new statuses to the exclusion list.

The existing `serial_numbers` collection is reconciled with `stock_units`: instead of a parallel unlinked CRUD, serial tracking becomes an optional property (`serial`) on `stock_units`. The legacy `serial_numbers` CRUD endpoints remain for backward compat but are bridged: a `serial-bind` event on a stock_unit sets `stock_units.serial` and writes a reconciling `serial_numbers` row.

The `base_bank_targets` collection stores a per-store (or entity/global) target quantity per grid cell (planogram slot, reader power step, CL power step, CL colour). `GET /inventory/replenishment` computes `required = base_bank - in_hand` per cell. TASKMASTER consumes this for auto-reorder.

## Acceptance tests (INTENT-LEVEL; a hollow shell must FAIL)

1. **Ledger is append-only.** After 5 events on a unit, `GET /items/{stock_id}/events` returns exactly 5 rows in `event_seq` order. Attempt to `DELETE` or `PUT` any row returns 405. The `event_seq` values are strictly increasing integers with no gaps within a single unit's history.

2. **Illegal transition refused.** `POST /items/{unit_id}/quarantine` on a SOLD unit returns HTTP 409. No `item_events` row is inserted. `stock_units.status` remains SOLD. `POST /items/{unit_id}/quarantine/release` on an AVAILABLE unit (never quarantined) returns 409. State machine is pure and unit-tested independently.

3. **Concurrency-safe sell.** Seed one AVAILABLE unit. Fire two simultaneous `SELL` events (via `record_event_atomic`). Exactly one returns `ok=True`; the other returns `ok=False` (409 from the CAS). `item_events` contains exactly one SELL row. `stock_units.status` is SOLD. On-hand for the product is 0.

4. **Quarantine drops on-hand immediately.** Seed product with 2 AVAILABLE units. Call `POST /items/{unit_1}/quarantine`. `GET /inventory/products/{product_id}/stock` returns `on_hand=1` (not 2). The QUARANTINE unit does NOT appear in `find_available` result. POS attempt to sell the quarantined unit returns 409.

5. **Quarantine round-trip.** AVAILABLE -> QUARANTINE via `quarantine.in` event. Release with `RESTOCK` -> QUARANTINE -> AVAILABLE via `quarantine.out` event; on-hand rises back to original. Release with `DAMAGE` -> QUARANTINE -> DAMAGED; on-hand does NOT rise. Release with `RTV` -> QUARANTINE -> RTV; on-hand does NOT rise. Each release leaves exactly two `item_events` rows (the original `quarantine.in` + the `quarantine.out`).

6. **Blind-count under-audit.** Start a cycle-count session: all in-scope units' statuses become UNDER_AUDIT and each gets an `audit.flag` event. `find_available` for those product IDs returns 0. POS cashier adding any UNDER_AUDIT unit to cart receives a warning (soft-block per DECISIONS §2 blind-count). On reconcile-complete: shrinkage units get VOID events, remaining units get `audit.clear` events and return to AVAILABLE. On-hand matches the accepted physical count.

7. **Transfer in-transit truth.** Ship 3 units of a product from Store A to Store B. Source `find_available` for that product at Store A drops by 3 immediately. The 3 units have `item_events` rows of type `transfer.ship`. Receive 2 of the 3 at Store B: destination `find_available` rises by 2; the re-homed units have `transfer.receive` events and their `store_id` is now Store B. The 1 unreceived unit remains TRANSFERRED (not counted in either store's on-hand). Repeat ship call (double-ship) is a no-op (idempotency flag).

8. **Serial-bind + return mismatch hard-block.** Mark a SKU as `serial_required=True` on the product. Receive 1 unit; GRN mints a AVAILABLE stock_unit without a serial. POS sell of that unit without a prior `serial-bind` is blocked (409). After `serial-bind` with serial "SN-001", the sell proceeds. On a return: scan serial "SN-999"; return endpoint compares to `stock_units.serial="SN-001"`; mismatch -> HTTP 409 with `reason=SERIAL_MISMATCH`; refund is NOT processed. With manager PIN override (E4 PIN): return proceeds; event carries `override_by` actor.

9. **Base-Bank replenishment math.** Insert `base_bank_targets` for store S1: grid READERS, cell "+2.00", base_bank=10. Stock 6 AVAILABLE units of the reader at +2.00 for S1. `GET /inventory/replenishment?store_id=S1&grid=READERS` returns `[{cell_key:"+2.00", base_bank:10, in_hand:6, required:4}]`. Power cell "-1.75" does not snap; cell "-1.75" does not equal "-1.50". Use `format_power` to normalise to "+2.00" — assert the cell key returned is the normalised form.

10. **E2 scope hierarchy.** Create a GLOBAL base_bank target for cell "+3.00" = 8. Create a STORE-scoped target for the same cell at store S2 = 5. `GET /inventory/replenishment?store_id=S2&grid=READERS` uses 5 (STORE wins). A second store S3 with no STORE target uses 8 (GLOBAL fallback).

11. **Colour-flag migration.** The legacy green/amber/red status vocabulary (any lowercase status, any variant of "in_stock", "available", "reserved", "void") maps to the canonical `StockState` enum on write. No `item_events` row carries a raw colour string. The rollup exclusion test (intent test #4) passes even when the unit was minted with lowercase "available".

12. **On-hand rollup excludes all non-sellable statuses.** Seed one unit each in QUARANTINE, UNDER_AUDIT, BLIND_COUNT, TRANSFERRED, SOLD, VOID, DAMAGED, RTV. `_on_hand_by_product` returns 0 for that product. `find_available` returns 0. The non-moving stock report does not include the product. The CL stock report does not count QUARANTINE/UNDER_AUDIT units.
