## Intended behavior (full intent)

E6 is the single config-driven outbound rail for all recurring/triggered customer messages. A `reminder_rules` document describes who to contact (via a `segment_key`), when (CRON or EVENT trigger), on which channel, with which template, and whether a voucher is attached. At evaluation time the rule resolves its audience through the existing segment resolvers, then passes each recipient through an ordered gate stack — consent (via `is_opted_out`), quiet-hours (promotional only, via `in_quiet_hours`), voucher pre-check, and the cross-rule 30-day frequency cap (3 messages / customer / 30 days per DECISIONS §2 #10) — before queuing through `send_notification`. E6 is the policy/eligibility layer only; it does not introduce a second sender, a second consent store, or a second clock.

Every recurring reminder type — Rx-expiry, birthday, win-back, CL-reorder, churn alert, lookbook, NPS/feedback, FU-Due-Today — becomes a `reminder_rules` row instead of a bespoke code branch. Adding a new reminder type is schema authoring, not engine surgery.

Feature #46 ("Configurable Reminders") is fully satisfied when: (a) each rule type has a UI-editable config row (template, timing, active toggle); (b) the gate stack is correctly enforced; (c) the 30-day cap is shared across all rules including manual campaigns; (d) FU-Due-Today drives outbound for WHATSAPP/SMS modes and creates staff tasks for CALL/IN-PERSON modes.

**Phase split (ENGINES.md Wave 0b / Wave 3):** Wave 0b ships only the transactional slice — `pool_otp` collection + `send_pool_redemption_otp`/`verify_pool_redemption_otp` + `comms_ledger` + `check_frequency_cap`/`record_outbound`. ## Backend (endpoints + services + which ENGINE calls)

**Service: `backend/api/services/reminder_rail.py`**

`evaluate_rule(db, rule, *, now=None, dry_run=False)`:
- Calls `campaign_segments.resolve_segment(db, rule["segment_key"], store_id=rule.get("store_id"), params=rule.get("segment_params") or {})` to get audience rows.
- For each recipient: calls `passes_gates(db, rule, recipient, now=now)`.
- If passes and not dry_run: calls `send_notification(...)` (notification_service.py:107) with `rule_id` stamped; then `record_outbound(db, customer_id, ...)`.
- If rule has `voucher_template` and recipient passes: calls `vouchers.issue_voucher`-equivalent, injects `{voucher_code}` into template variables.
- For `fu_due_today` segment recipients with `mode` in `{CALL, IN_PERSON}`: creates a staff task instead of queuing a message.
- Returns a result dict with counts: resolved, skipped_consent, skipped_freqcap, skipped_quiet, skipped_no_phone, queued, voucher_minted, errors.

`passes_gates(db, rule, recipient, *, now=None) -> (bool, reason|None)`:
- Gate 1 — consent: if not `rule["is_transactional"]`, calls `marketing.is_opted_out(customer_id, db)` (marketing.py:1463). Fail → ("consent", reason).
- Gate 2 — quiet-hours: if not `rule["is_transactional"]`, calls `quiet_hours.in_quiet_hours(now)` (quiet_hours.py:83). Fail → schedules `scheduled_for = next_quiet_end_utc_iso()` instead of None, still queues (defer, not drop). Returns pass with deferred flag.
- Gate 3 — freq-cap: if not `rule["freq_cap_exempt"]` and not `rule["is_transactional"]`, calls `check_frequency_cap(db, customer_id, now=now)`. Fail → ("freqcap", reason).
- All gates pass → (True, None).

`check_frequency_cap(db, customer_id, *, now=None, window_days=30, max_msgs=3, category="MARKETING") -> bool`:
- Counts `comms_ledger` rows for `customer_id` where `sent_at >= (now - 30d)` and `category != "OTP"` and `category != "SERVICE"`. Returns True if count < max_msgs.
- The `max_msgs=3` default is locked (DECISIONS §2 #10). E2's `comms.cap_per_customer_30d` key overrides this when E2 is live; fall back to 3 if key absent.

`record_outbound(db, customer_id, *, channel, category, rule_id=None)`:
- Inserts one `comms_ledger` row. Called after successful `send_notification` queue, before returning from `evaluate_rule`.
