## Intended behavior (full intent)

A reusable, PIN-gated, time-limited, store-scoped, audit-chained maker-checker engine that every domain consumes instead of ad-hoc approval logic. The engine's purpose: give every sensitive action a real audit trail with a named approver who put a credential on it, time-bound so approvals do not drift open, and protected against double-use (one approval = one action, guaranteed by the same atomic pattern as voucher redemption).

**Full lifecycle:**

1. **Maker requests** ‚Äî any authenticated user calls `request_approval(action_type, amount, context, reason)`. Engine resolves the required tier from configurable thresholds (auto / admin / super), records a `REQUESTED` doc in `approval_requests` with a 60-minute `expires_at`, sends an in-app bell notification and (gated by `DISPATCH_MODE`) a WhatsApp to eligible approvers, deduplicates identical live requests.

2. **Approver acts (PIN-gated, atomic)** ‚Äî an approver with the right role AND a set `approval_pin_hash` calls `approve(request_id, pin)`. The engine verifies the PIN against bcrypt hash, checks expiry, checks role eligibility, checks store-scope, and ‚Äî for `journal_entry` ‚Äî checks approver ‚âÝ maker. The state transition from `REQUESTED` to `APPROVED` is a single `find_one_and_update` whose filter simultaneously guards `status==REQUESTED`, `expires_at > now`, and `consumed==False`. The same op mints the `approval_token`. If two approvers race, exactly one wins; the other gets 409. A wrong PIN increments a `pin_attempts` counter on the user doc; five failures in 15 minutes return 423 Locked.

3. **Maker consumes (atomic single-use)** ‚Äî the maker passes the `approval_token` (or `request_id`) to `consume_approval(...)`. A single `find_one_and_update` guards `status==APPROVED`, `consumed==False`, `expires_at > now`, `action_type` matches, `amount` (if supplied) is within the approved amount. Exactly one consume wins; a second gets `already_consumed`. Only after ok does the caller perform the real DB change (discount saved, refund executed, JE posted).

4. **Expiry** ‚Äî `TASKMASTER`'s 5-min tick runs `expire_stale()`, which bulk-updates `REQUESTED` docs whose `expires_at < now` to `EXPIRED` and writes one audit row each. Every individual read also lazy-checks and flips overdue rows. No Mongo TTL index (that would silently delete auditable records).

5. **Audit** ‚Äî every transition (requested / approved / rejected / expired / consumed / pin_set / pin_cleared) writes a hash-chained row to `audit_logs` via `AuditRepository.create` with `entity_type="approval_request"`. PIN values and hashes never appear in any audit row.

**Action types supported at launch:** `discount_override`, `refund`, `journal_entry` (maker-checker enforced), `profile_merge`, `petty_cash`, `endless_aisle`, `rtv`.

**Tiers (default; overridable via E2 when E2 lands):**
- `auto` (< ‚Çπ500): STORE_MANAGER / AREA_MANAGER / ADMIN / SUPERADMIN
- `admin` (> ‚Çπ2,000): ADMIN / AREA_MANAGER / SUPERADMIN
- `super` (> ‚Çπ10,000): SUPERADMIN only
- `serial_mismatch`: always `admin` tier, `required_tier` override explicit

**INTENT-FIDELITY gate:** the existing `ApprovalWorkflows.tsx` settings panel is a configuration shell. The new `ApprovalInboxPage` and `PINApproveModal` are the live action surfaces. A hollow implementation that just sets `status=APPROVED` without a PIN, without a single atomic `find_one_and_update`, without a real `approval_token`, without TTL enforcement, without an audit row ‚Äî FAILS every acceptance test below.

## Backend (endpoints + services + which ENGINE calls)

### Service: `backend/api/services/approvals.py`

Class `ApprovalEngine` ‚Äî modelled on `ProposalStore` (proposals.py:106-558):

- `__init__(db=None)` ‚Äî fail-soft; `db=None` = all reads return empty, writes are no-ops
- `_coll()` / `_audit_coll()` ‚Äî mirror proposals.py:124-144 collection access pattern
- `_write_audit(action, request_doc, actor)` ‚Äî calls `AuditRepository.create` with `entity_type="approval_request"`, `before_state`/`after_state` snapshots, never includes PIN or hash
- `_resolve_tier(amount, action_type, store_id, entity_id)` ‚Äî reads `_DEFAULT_TIERS` constant (`auto<500 / admin>2000 / super>10000`); seam for E2 `resolve_setting` when E2 lands
- `_tier_to_roles(tier)` ‚Äî maps tier to `required_roles` list
- `request(...)` ‚Äî dedupes via `dedupe_key` lookup on `{status:REQUESTED}`; inserts doc; calls in-app bell write + `notification_service.send_notification` (gated by `DISPATCH_MODE`); returns `{ok, request_id, status, required_roles, expires_at}`
- `approve(request_id, approver_user_id, approver_roles, pin)`:
  1. Check `pin_attempts` throttle (5 attempts / 15 min window) ‚Äî 423 if locked
  2. Verify PIN via `verify_password(pin, user.approval_pin_hash)` (auth.py:232); increment `pin_attempts` on fail; reset on success
  3. Single `find_one_and_update` with filter `{request_id, status:"REQUESTED", expires_at:{$gt:now()}, consumed:false}` + `$set {status:"APPROVED", approval_token:"APT-<uuid>", reviewed_by, reviewed_at}` + `return_document=ReturnDocument.AFTER` ‚Äî mirrors vouchers.py:217
  4. If no doc matched: distinguish expired vs already-reviewed via a follow-up read (mirrors `_redeem_failure_reason` in vouchers.py:264-283) ‚Äî 410 expired / 409 conflict
  5. For `maker_checker==True`: check `approver_user_id != requested_by` before the update ‚Äî 403 `cannot_approve_own`
  6. Check role in `required_roles` ‚Äî 403 `insufficient_tier`
  7. Check store-scope (STORE_MANAGER: `approver_stores` must include `request.store_id`; HQ roles bypass)
  8. Write audit row; return `{ok:True, approval_token, ...}`
- `reject(request_id, approver_user_id, approver_roles, pin, reason)` ‚Äî same PIN/role/scope/expiry guards; `find_one_and_update` to `REJECTED`; audit row
- `consume_approval(request_id=None, approval_token=None, consumed_by, action_type, amount=None)`:
  - Single `find_one_and_update` filter: `{$or:[{request_id},{approval_token}], status:"APPROVED", consumed:false, expires_at:{$gt:now()}}` + `$set {consumed:true, consumed_at, consumed_by, status:"CONSUMED"}`
  - Post-match checks (on the returned doc): `action_type` matches; `amount <= approved_amount` if supplied
  - Returns `{ok:True, request:{...}}` or `{ok:False, error:"already_consumed|expired|not_approved|amount_exceeded|action_mismatch"}`
  - Audit row on success
- `get(request_id)` ‚Äî single fetch; lazy-expires if overdue
- `list_inbox(approver_roles, store_ids, status, limit)` ‚Äî query on `(status, required_roles, store_id)`; HQ roles get all stores
- `list_mine(requested_by, limit)` ‚Äî query on `(requested_by, status)`
- `expire_stale()` ‚Äî `update_many({status:"REQUESTED", expires_at:{$lt:now()}}, {$set:{status:"EXPIRED"}})` + one audit row per expired doc; returns count

**PIN helpers (module-level):**
- `set_approver_pin(db, user_id, pin, set_by)` ‚Äî validates PIN (4-6 digits); `hash_password(pin)` ‚Üí `users.approval_pin_hash`; audit row (`pin_set`); clears `pin_attempts`

- `clear_approver_pin(db, user_id, cleared_by)` ‚Äî unsets hash; audit row (`pin_cleared`)

## Engine + CORRECTIONS folded

**CORRECTIONS.md ¬ßP1/E4 ‚Äî fully folded:**

1. `approve()` is a single atomic `find_one_and_update({request_id, status:"REQUESTED", expires_at:{$gt:now()}, consumed:false})` that mints the `approval_token` in the same operation (not a read-then-write). This prevents the two-approves-one-token race.

2. PIN brute-force throttle: `pin_attempts: {count, window_start}` sub-doc on `users`. Five wrong PINs within 15 minutes ‚Üí 423 Locked. Counter resets on correct PIN or when `now > window_start + 15 min`. Implemented in `approvals.py` using a `find_one_and_update` on the user doc (Mongo, not Redis) so it works even when Redis is absent. Acceptance test required (see test #2 below).
