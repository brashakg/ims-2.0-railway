## Intended behavior (full intent)

A single typed policy registry (`policy_registry.py`) declares ~30 named keys with their types, defaults, validation rules, env fallbacks, and which scope levels allow overrides. A resolver service (`policy_engine.py`) answers `get_policy(key, scope)` with the most-specific value present: store override beats entity override beats global override beats registry-code default beats env var.

Every engine that previously embedded a number inline instead calls `get_policy(key, {"store_id": sid})`. The owner can open Settings → Policies, pick a key (e.g. `cash.variance.block`), and set a different value for "Store: Bokaro" without touching code or redeploying. A STORE_MANAGER can tune store-scoped keys for their store only; global/entity writes require ADMIN or above.

The registry is code-versioned (defaults survive a fresh DB), the override DB is sparse (only set keys stored), and the FE renders the full form from the server-published schema — adding a new key in `policy_registry.py` surfaces a control in Settings with no FE change.

Every write is hash-chain-audited via `AuditRepository.create` (the single sink per ENGINES.md:40) and the before/after diff surfaces in the existing Activity Log via `_audit_changes` (settings.py:1952) with zero new FE work.

Resolution order inside `get_policy`:
1. Cache lookup: key `policy:<scope_id>:<key>` (TTL_LONG = 900 s from cache.py).
2. DB read: `policy_settings.find_one({"_id": "store:<store_id>"})` → `values.get(key)`.
3. If store miss, resolve `entity_id` via `stores` collection (`store.entity_id`, required field per stores.py:53), then `policy_settings.find_one({"_id": "entity:<entity_id>"})`.
4. If entity miss, `policy_settings.find_one({"_id": "global"})`.
5. Registry code default from `policy_registry.REGISTRY[key].default`.
6. Env fallback from `policy_registry.REGISTRY[key].env` via `os.getenv`.
7. `default` arg if provided, else `KeyError`.

The store→entity lookup adds one `stores` collection read per cold cache miss. Acceptable: 6 stores total, result cached. No DB read on warm cache hit.

Cache invalidation in `set_policy`: `cache.delete(f"policy:{scope_addr}:{key}")`. On global write, also invalidate all entity/store keys for that policy key (pattern delete, or delete known scope_ids — cardinality is 10, so enumerate).

`set_policy` validation sequence:
1. Key exists in registry (else `KeyError`).
2. Scope level is allowed for this key (`PolicySpec.scopes`).
3. Caller's role is in `PolicySpec.write_roles` (else `403`).
4. For store-scope writes: caller's `store_ids` includes `scope["store_id"]` (store ownership check, same pattern as settings.py:2302-2307).
5. Type/min/max/enum/regex validation (reject with descriptive error).
6. Special guard for `pricing.category_caps.*` keys: new value must be `<=` the `pricing_caps.py` constant (can only lower, never raise luxury brand caps via E2).
7. Encrypt if `PolicySpec.secret`.
8. Upsert `policy_settings` doc.
9. `AuditRepository.create({action:"policy_update", entity_type:"policy_setting", entity_id:key, before_state:{key: old_val}, after_state:{key: new_val}, store_id:...})` — single audit sink per ENGINES.md:40.
10. Cache invalidate.
11. Return `get_effective(key, scope)`.
