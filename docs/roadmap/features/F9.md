## Intended behavior (full intent)

External-lab lenses (those with `lens_status=ORDERED`, meaning the lens was sent to an external vendor like Zeiss/Essilor/Alifnoor/GKB/Vinod) arrive at the store with a **Delivery Challan** — a physical receipt document the vendor sends with the goods. The vendor's tax invoice comes later (monthly or fortnightly). Currently there is no record of the DC in IMS; the accountant matches invoices to mental memory of what arrived, creating a cash leakage risk (phantom payments, lost lenses).

**Intent: make the DC the mandatory checkpoint between physical lens arrival and workshop work, and between goods movement and accounting.**

1. **DC logging**: a Store Manager logs the DC when lenses physically arrive, before handing them to workshop. This creates a GRN record with `grn_subtype=DELIVERY_CHALLAN`, `dc_number`, and `dc_date`. The existing GRN accept path mints barcoded `stock_units` identically. `vendor_invoice_no` is NOT required for a DC (it arrives later).

2. **Workshop HARDLOCK**: if a workshop job's `lens_status=ORDERED` (external lab), creating or advancing it to `IN_PROGRESS` requires at least one accepted, unmatched DC covering that lens SKU (`product_id`) at that store. No DC = 422 with a clear message. Override (ADMIN+) requires `override_reason` field, audited. A config flag `REQUIRE_DC_FOR_WORKSHOP` (default `true` in prod, `false` in dev/test) enables a grace period without redeploy. **The HARDLOCK applies only to jobs where `lens_status=ORDERED` — in-house stock jobs are exempt.**

3. **Month-end DC-to-invoice tally**: the accountant selects a set of open (unmatched) DCs for a vendor within a 30-day window, generates a consolidated draft invoice, reviews it, and books the bulk invoice. The engine (`dc_bulk_match`) aggregates accepted qty across all selected DCs per SKU, compares to the vendor's billed qty, returns `MATCHED` / `ON_HOLD_EXCEPTION` verdict. Exception = auto-hold (same behavior as existing 3-way match). On book, every linked DC is stamped `dc_matched=true` and `linked_bulk_invoice_id`.

4. **Audit immutability**: DC accept, bulk-invoice match, HARDLOCK trigger (including override) all write `AuditRepository.create` rows. No event is ever updated.

5. **Backward compatibility**: existing GRNs with no `grn_subtype` behave as `STANDARD`. The HARDLOCK only fires for jobs created after a configurable `dc_hardlock_from_date` (defaults to the deploy date), never retroactively blocking existing pending jobs.

## Acceptance tests (INTENT-LEVEL; a hollow shell must FAIL)

1. **DC logging blocks invoice-only GRN**: POST `/grn` with `grn_subtype=DELIVERY_CHALLAN` and no `vendor_invoice_no` succeeds (201). POST `/grn` with no `grn_subtype` and no `vendor_invoice_no` fails (422). A hollow shell that ignores `grn_subtype` cannot pass both.

2. **DC number uniqueness**: POST two DCs with the same `(vendor_id, dc_number, store_id)` — the second returns 409. A different store or different dc_number succeeds. The uniqueness is per store (not just per vendor) because vendors may use the same DC number across branches.

3. **HARDLOCK fires for external-lab lenses**: create a workshop job where `lens_details.lens_status=ORDERED` with no accepted DC for the lens product at that store — returns 422 with `DC_HARDLOCK`. Create the same job where lens_status is NOT `ORDERED` (in-house stock) — succeeds (201). A shell that always returns 201 fails test 1; a shell that always 422s fails test 2.

4. **HARDLOCK respects cutover date**: a job with `lens_status=ORDERED` created BEFORE `dc_hardlock_from_date` succeeds even with no DC. Created ON or AFTER the date → 422 (or proceeds with override). A shell that ignores the date fails this.

5. **HARDLOCK override is audited and gated**: ADMIN submits job create with no DC but `override_reason="Emergency — DC in transit"` → 201. STORE_MANAGER submits the same → 422 (no override privilege). An `audit_logs` row with `action=dc_hardlock_override` exists after the ADMIN create. A shell that lets any role override fails the role check; a shell that does not write the audit row fails the immutability check.

6. **DC accept mints stock identically to standard GRN**: POST `/grn/{dc_id}/accept` mints `stock_units` with `source_type="GRN"`, `source_id=dc_id`, barcode generated. The on-hand count for the product at that store increases by accepted_qty. A shell that skips minting for DCs fails the on-hand check.

7. **Bulk-invoice DC aggregation**: given DCs D1 (50 units product P1), D2 (30 units P1, 20 units P2), a `GET /from-dcs?dc_ids=D1,D2` draft shows P1 qty=80, P2 qty=20. A shell returning empty lines fails.

8. **DC-to-invoice match MATCHED**: book a bulk invoice matching the exact D1+D2 quantities — `dc_match_status=MATCHED` on the bill, `dc_matched=true` on D1 and D2, `linked_bulk_invoice_id` set. After booking, a second attempt to link D1 to another invoice returns 409 (D1 already matched). A shell that does not flip `dc_matched` fails the re-link guard.

9. **DC-to-invoice match ON_HOLD_EXCEPTION (auto-hold)**: book a bulk invoice where vendor billed 100 units of P1 but DCs show only 80 accepted (25% variance, above 5% tolerance) — `dc_match_status=ON_HOLD_EXCEPTION`, invoice status = `ON_HOLD_EXCEPTION`. Invoice is still persisted (AP liability recorded) but flagged. A shell that silently posts as MATCHED fails; a shell that hard-blocks and does not persist the AP liability also fails (the existing 3-way match books and flags, never hard-blocks).

10. **Period lock on earliest DC date**: DCs from month M are period-locked. Attempting to book a bulk invoice linking those DCs returns 422 (period locked), even if the invoice_date is in the current open month. A shell that checks only `invoice_date` fails.

11. **REQUIRE_DC=false bypasses hardlock**: set `require_dc_for_workshop=false` in purchase_settings. Workshop job with `lens_status=ORDERED` and no DC → 201. Restore flag to true → 422 again. A shell that hard-codes the block fails the flag-off test.

12. **Backward compatibility**: existing GRNs in the DB with no `grn_subtype` field are treated as STANDARD. Existing `GET /grns` without filters returns them. Existing `GET /purchase-invoices` returns invoices without `dc_match_status` as status `N_A` (or null). No existing test breaks.
