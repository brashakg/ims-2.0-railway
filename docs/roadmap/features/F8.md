## Intended behavior (full intent)

F8 closes the procurement accountability loop: every quantity or price discrepancy between what was ordered, what arrived, and what was invoiced must be recorded, investigated, and resolved — not silently absorbed. Backordered goods must be surfaced before they become forgotten liabilities.

**PO vs GRN variance report** — a live, filterable table on the Purchase page showing every open/partial PO with its per-line variance: ordered qty, received qty, open (outstanding) qty, accepted qty, rejected qty, line `variance_status` (SHORT/OVER/EXACT/UNMATCHED), days since PO expected date. Aged lines (past `expected_date`) shown in a distinct visual state — not a color flag, but an explicit `aging_status` enum field (`ON_TIME | OVERDUE | CRITICALLY_OVERDUE`).

**Aged-backorder TASKMASTER sweep** — every 5-minute TASKMASTER tick scans `purchase_orders` for POs with status `SENT|ACKNOWLEDGED|PARTIALLY_RECEIVED` and `expected_date` in the past. For each open line with un-received qty still outstanding, if no active task exists (dedupe key `backorder:{po_id}:{product_id}`), a P2 task is created. If the PO has been overdue for 14+ days and the task is still open, TASKMASTER escalates it to P1 and upgrades the task title to reflect urgency. This is Tier 1 auto-act (task creation is fully reversible).

**Admin "Dismiss with justification"** — an ADMIN/ACCOUNTANT can dismiss a variance/backorder task from the Purchase page without creating a debit note. Dismiss requires a mandatory `reason` string (min 10 chars) written to an audit row (`audit_logs` via `AuditRepository.create`). The dismissed task moves to status `DISMISSED` (not COMPLETED — so SLA/escalation logic does not treat it as genuine resolution). Dismissal is recorded on the PO doc: `dismissed_variances[]` list of `{po_id, product_id, reason, dismissed_by, dismissed_at}` via a single `find_one_and_update`.

**Debit-note prompt on dismiss with invoice** — when the variance being dismissed involves an accepted GRN that is linked to a booked purchase invoice (`grn_id` and `bill_id` both present), the dismiss API response includes `debit_note_suggested: true` with the draft amounts (invoiced minus accepted). The frontend renders a banner: "A debit note of Rs. X may be appropriate — create one?" linking to the existing `POST /{vendor_id}/debit-notes` flow. This is a prompt only — the operator can ignore it.

**Match detail on invoice tab** — the existing invoice list gains a "Match" column showing `MATCHED | ON_HOLD | OVERRIDE` badges. Clicking a held invoice opens a read-only side panel with the per-line variance table (ordered/received/invoiced qty, price variance %). The existing `GET /{id}/match` and `POST /{id}/approve-exception` endpoints are already built; the UI is the delta.

## Backend (endpoints + services + which ENGINE calls)

No shared engines (E1–E6) are required for this feature. It reads existing PO/GRN data and writes via single-doc mutations.

| Endpoint | Method | Auth | Description |
|---|---|---|---|
| `/vendors/variance-report` | GET | ADMIN, ACCOUNTANT, STORE_MANAGER | Per-line variance: open_qty, days_overdue, aging_status |
| `/vendors/purchase-orders/{po_id}/dismiss-variance` | POST | ADMIN, ACCOUNTANT | Dismiss a variance line with justification; debit-note prompt |

**`GET /vendors/variance-report`** logic:
1. `po_repo.find_pending()` — returns SENT/ACKNOWLEDGED/PARTIALLY_RECEIVED POs (already exists, `vendor_repository.py:74-78`).
2. Filter by `store_id` query param if provided (`delivery_store_id` field).
3. For each PO, fetch its GRNs from `grn_repo` (direct collection query by `po_id`).
4. Call `po_variance_engine.open_qty_per_line(po, grns)` — pure, returns per-product `{ordered_qty, received_qty, open_qty, accepted_qty, rejected_qty, variance_status, days_overdue, aging_status}`.
5. Filter out lines where `open_qty == 0` AND `variance_status == "EXACT"` (nothing to show).
6. Return paginated list (default limit 50).
7. Fail-soft: DB unavailable → empty list.

**`POST /vendors/purchase-orders/{po_id}/dismiss-variance`** logic:
1. Verify PO exists (`po_repo.find_by_id`). 404 if not.
2. Validate `reason` length >= 10 chars. 400 if not.
3. Compute `debit_note_suggested`: if both `grn_id` and `bill_id` in request body, query `vendor_bills` for the invoice, fetch GRN accepted_qty, invoice invoiced_qty. If invoiced_qty > accepted_qty for the given product_id, set `debit_note_suggested=True`, `suggested_amount = (invoiced_qty - accepted_qty) * invoice_unit_price`.
4. Build `entry` dict (as above in data model).
5. `purchase_orders.find_one_and_update({"po_id": po_id}, {"$push": {"dismissed_variances": entry}, "$set": {"updated_at": now}})` — single doc, single collection.
6. `AuditRepository.create({action: "po_variance_dismiss", ...})` — fail-soft.
7. Return `{dismissed: True, debit_note_suggested: bool, suggested_amount?: float, vendor_id: str}`.

**`po_variance_engine.py`** (new pure service):
- `open_qty_per_line(po: dict, grns: list[dict]) -> list[dict]` — sums `accepted_qty` per product across ACCEPTED GRNs, computes `open_qty = ordered_qty - received_qty`, calls `classify_grn_line_variance()` for label, computes `days_overdue` from `expected_date`, assigns `aging_status` enum: `ON_TIME` (not past expected), `OVERDUE` (1-13 days past), `CRITICALLY_OVERDUE` (14+ days past).
- `aged_backorder_tasks_needed(pos, grns_by_po, now, threshold_days=14) -> list[dict]` — returns task specs for TASKMASTER. Skips dismissed lines (checks `dismissed_variances[]` on each PO).

**TASKMASTER `_sweep_aged_backorders()`** — new async method, called from `_do_background_work()` after `_draft_reorders()`. Tier 1 auto-act. Calls `po_variance_engine.aged_backorder_tasks_needed()`, then for each spec calls `create_system_task()` (already imported in vendors.py; replicate import). For tasks already created and still open past 14 days, calls `tasks.find_one_and_update({"source_ref": ref, "status": {"$in": ["OPEN","IN_PROGRESS"]}}, {"$set": {"priority": "P1", "updated_at": now}, "$push": {"history": {...}}})` — single document update.
