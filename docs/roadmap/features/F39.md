## Intended behavior (full intent)

Each morning before store opening, MEGAPHONE scores every active customer per store across six weighted signals (Rx expiry, CL refill, birthday, purchase anniversary, winback, tags), produces a ranked list of exactly 15 cards, and writes it to `nba_scores` (TTL-expires at IST midnight). The top 2 slots are always reserved for VIPs (LTV >= ₹15,000 single-order OR `tags` contains "VIP"); slots 3-15 are pure score-ranked.

Associates open "Daily Calls" in the rail and see their store's 15 prioritised customers for today. Each card shows: customer name/mobile (tap-to-call), one headline reason ("Rx expires in 5 days — book eye test"), up to 2 sub-headlines ("Last visited 4 months ago", "Gold tier"), loyalty-tier chip, and any tags. Associates mark each card Done (with mandatory outcome notes, min 10 chars) or Skip (with a reason from a fixed enum). Done/Skip actions write to the pre-created `follow_up` doc (type=`nba_call`) via the existing `PATCH /follow-ups/{id}/complete`.

Score value is internal-only; rank is what associates see. A dismissed customer does not reappear the same day. Historic completions persist in `follow_ups` as the durable audit trail; `nba_scores` expires daily.

Staff may suggest tags on customers they serve; Store Managers approve or reject suggestions before they become visible on the customer record. Tags feed next-day NBA scoring. Tags must not contain phone numbers, email addresses, or GSTINs.

Given 100 customers in a store with varying signals, when `GET /api/v1/crm/nba/{store_id}` is called, then the response contains exactly 15 cards (or fewer if the store has fewer than 15 scoreable customers, but never more than 15).

**T4 — Score value hidden from API response:**
Given a scored `nba_scores` doc that has a `score` field on each card, when `GET /crm/nba/{store_id}` is called, then no card in the response JSON contains a `score` key. (Asserts intent: gaming-prevention.)

**T5 — Dismiss removes card from list, writes audit:**
Given a card for customer X in today's NBA list, when `POST /crm/nba/{store_id}/dismiss {customer_id: X, reason: "no_answer"}` is called, then: (1) a subsequent `GET /crm/nba/{store_id}` returns 14 cards without customer X; (2) the `follow_ups` doc for the pre-created `nba_call` has `status=skipped, outcome=no_answer`; (3) `audit_logs` contains an entry with `action="nba.dismissed", entity_id=X`.

**T6 — Complete requires outcome_notes >= 10 chars:**
When `POST /crm/nba/{store_id}/complete {customer_id: X, outcome_notes: "short"}` is called (8 chars), then the API returns 422. When called with notes of 10+ chars, it returns 200 and the follow-up doc has `status=completed, notes=<those notes>`.

**T7 — Tag suggest-approve-visible workflow:**
Given a SALES_STAFF user who calls `POST /customers/{id}/tags/suggest {tag: "Zeiss fan"}`, then: (1) `tag_suggestions` has a doc with status="pending"; (2) `GET /crm/nba/{store_id}` does NOT include "Zeiss fan" in the customer's `tags` array yet; (3) after a STORE_MANAGER calls `POST .../approve`, `customers.tags` contains "Zeiss fan" and the next `_score_nba_sync` run reflects it.

**T8 — Tag PII stripping:**
When `PATCH /customers/{id}/tags {tags: ["call 9876543210", "john@example.com", "GSTIN 22AAAAA0000A1Z5", "Cartier fan"]}` is called by a STORE_MANAGER, then only `"Cartier fan"` is saved (the others are stripped/rejected). Response contains `tags: ["Cartier fan"]`.

**T9 — Staff cannot approve tags, only suggest:**
When a SALES_STAFF user calls `POST /customers/{id}/tags/suggestions/{sid}/approve`, the API returns 403.

**T10 — MEGAPHONE idempotency:**
Given today's `nba_scores` doc already exists for a store, when `_score_nba_daily()` runs a second time for the same store and date, then no new `nba_scores` doc is written and no duplicate `follow_up` docs are created (existing docs remain unchanged).

**T11 — Fallback scoring when MEGAPHONE hasn't run:**
Given no `nba_scores` doc exists for today, when `GET /crm/nba/{store_id}` is called, then the response still contains a valid cards list (up to 15) computed synchronously. Response `generated_at` is approximately now (not a stale timestamp).

**T12 — RBAC: ACCOUNTANT gets 403 on NBA list:**
When a user with role ACCOUNTANT calls `GET /crm/nba/{store_id}`, the response is 403. This test MUST fail if the new RBAC policy row is missing from `rbac_policy.POLICY` (the existing coverage-lock test already enforces this once rows are present).

**T13 — Cross-store IDOR blocked:**
When SALES_STAFF at store A calls `GET /crm/nba/store_B_id`, they receive 403 (not a 200 with store B's data). Enforced by `validate_store_access`.
