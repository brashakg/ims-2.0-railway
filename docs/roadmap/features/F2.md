## Intended behavior (full intent)

A workshop order generates ONE disposable barcoded job card printed at intake. The barcode (Code128) encodes the job number. The card physically travels with the spectacles through every internal lab station. At each bench, the technician scans the card on a fixed tablet (station terminal); the system gates the scan to the correct station in sequence, advances the job's `current_station`, and records dwell time per station. The store manager sees a real-time floor map: how many jobs are at each station and which ones are overdue (exceeding that station's SLA).

When the job exits the final in-house station (DISPATCH = internal handoff to front desk), the system fires an automatic WhatsApp to the customer ("your eyewear is ready for pickup") via the existing `notify_ready` path — removing the manual step. The flow is:

**Station sequence (DECISIONS §3 "Disposable barcoded job cards"):**
```
ORDER CREATED
  → [Job card printed at INTAKE — Code128 barcode of job_number]
  → INTAKE scan → status: IN_PROGRESS, current_station: INTAKE
  → EDGING scan → current_station: EDGING  (lens cut to frame shape)
  → COATING scan → current_station: COATING  (AR/tint/PFX applied)  [skippable]
  → QC_LAB scan → current_station: QC_LAB  (bench QC; pass → continue; fail → QC_FAILED branch)
  → DISPATCH scan → current_station: DISPATCH, job status → READY
      [AUTO: WhatsApp "ready for pickup" fires here]
  → PICKUP scan (front desk) → status: DELIVERED
```

Station sequence is **store-configurable** (some stores skip COATING; pass-through labs may only have QC_LAB + DISPATCH). The forward-only gate rejects out-of-order scans with a loud structured error (never mutates state). Concurrent scans on the same job → `find_one_and_update` with `current_station` in filter → one succeeds, one gets 409. Every scan is immutable (append-only `scan_history`). Dwell time per station is computed on write (never client-supplied). The floor monitor board shows per-station queue counts and aging (green/amber/red by SLA %).

## Acceptance tests (INTENT-LEVEL; a hollow shell must FAIL)

Tests must assert the behavior, not the HTTP shape. A stub that returns 200 without mutating state must FAIL every test below.

1. **Forward-only gate enforced:** Scan job at COATING when `current_station=null` (never entered INTAKE) → `{ok:false, reason:"WRONG_STATION"}` AND `workshop_jobs.current_station` remains null AND `scan_history` is unchanged. A stub returning `{ok:false}` without checking DB passes the response check but would need a DB assertion to fail hollow.

2. **Happy-path station advance:** Create job (status=PENDING). Scan INTAKE → job.status=IN_PROGRESS, job.current_station=INTAKE, `scan_history` has one entry with `station=INTAKE`, `station_timestamps.INTAKE` is set (IST, within last 5s). Scan EDGING → current_station=EDGING, `station_dwell_ms.INTAKE` is a positive integer (elapsed since INTAKE). Status still IN_PROGRESS.

3. **DISPATCH auto-triggers status+notify:** Scan QC_LAB (current_station=QC_LAB) then scan DISPATCH → job.status=READY, current_station=DISPATCH, `ready_notified_at` is stamped (even in SIMULATED dispatch mode). A hollow shell that sets status without stamping notify OR that sets notify without setting READY fails.

4. **Concurrency safety:** Two parallel `POST /workshop/scan` requests for the same job at EDGING when `current_station=INTAKE` → exactly one succeeds (`ok:true`), one returns `ok:false` with reason `CONCURRENT_CONFLICT`. `scan_history` has exactly ONE EDGING entry. `station_dwell_ms.INTAKE` appears exactly once.

5. **Station sequence is store-configurable:** For a store with COATING deactivated (`is_active=false`), the sequence is INTAKE→EDGING→QC_LAB→DISPATCH. Scanning COATING at any point returns `{ok:false, reason:"UNKNOWN_STATION"}`. Scanning QC_LAB immediately after EDGING succeeds.

6. **Job card barcode is scannable:** Print `GET /workshop/jobs/{id}/label?type=traveler`, take `barcode_value` from the response. Simulating a keyboard-wedge scan of `barcode_value` to `POST /workshop/scan` with no `station` hint → resolves to the correct job (lookup by job_number). Assert `job_id` in response matches the job created.

7. **Real-time station queue:** After 3 jobs are scanned to EDGING and 1 is scanned on to QC_LAB, `GET /workshop/stations/EDGING/queue?store_id=X` returns exactly 3 jobs (not 4). The departed job does NOT appear.

8. **Auto-notify on DISPATCH is fail-soft:** If `agents.providers.send_whatsapp` raises an exception, the scan still returns `{ok:true, ...}` with `current_station=DISPATCH` and `job.status=READY`. The error is logged but does not roll back the station advance.

9. **Dashboard KPIs extended:** `GET /workshop/dashboard-kpis` response contains `per_station_counts` key with integer values for each station. After scanning 2 jobs to EDGING, `per_station_counts.EDGING == 2`. No existing key (`pending`, `overdue`, etc.) is removed or changed.

10. **Dwell time is server-computed, never client-supplied:** Attempt `POST /workshop/scan` with a body including a `station_dwell_ms` field → that field is ignored; the actual dwell is the server-computed elapsed time since `station_timestamps[previous_station]`. The stored value differs from any client-provided value.

11. **DELIVERED job rejects scan:** Job at status=DELIVERED → `{ok:false, reason:"TERMINAL_STAGE"}`. No state mutation. (Mirrors existing `scan_advance` terminal guard.)

12. **Code128 barcode on printed card:** `GET /workshop/jobs/{id}/label?type=traveler` → `barcode_value` is non-empty and equals `job.job_number`. The frontend renders a Code128 barcode element whose `data-value` attribute matches `barcode_value` (asserted in a render test against the `WorkshopJobCardPrint` component with `jsbarcode` mocked).
