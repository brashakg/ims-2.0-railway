## Intended behavior (full intent)

E1 is a pure service module (`money_guard.py`) — no FastAPI router of its own. It is the **single atomic guard** for every balance-mutating operation across the system. The intent, per DECISIONS.md and ENGINES.md, is:

1. **One primitive for all money moves.** Any code that reads a balance then writes it is a double-spend risk. E1 eliminates all read-modify-write paths by funnelling every mutation through a single `find_one_and_update` whose filter IS the guard. This is the same guarantee `redeem_voucher_atomic` provides today, generalized.

2. **Never raises on business failure.** Returns a typed `GuardResult` with `ok=False` and a machine-code `reason`. The caller decides HTTP status. Fail-soft on `db=None` (`reason="unavailable"`). Fail-CLOSED on collections lacking `find_one_and_update` for debits (`reason="no_atomic"`, balance untouched — exactly as `try_debit` does today).

3. **Idempotency built in.** Every mutation that can be retried (return reversals, refund credits, earn-on-order) carries an `idempotency_key`. A replayed call returns the original `txn_id` with `reason="duplicate"`. No double-credit, no double-debit on network retries.

4. **Audit on every successful move.** Each verb calls `AuditRepository.create(...)` after the atomic write. Audit failure is fail-soft and never blocks the money move, per the existing audit contract.

5. **Back-compat: Phase A only (this build).** `money_guard.py` ships as a unified facade over the three existing collections (`vouchers`, `loyalty_accounts`, `customers`) via adapter shims. The existing function signatures (`redeem_voucher_atomic`, `try_debit`, `try_debit_store_credit`) become 3-line shims calling E1 internals. Zero call-site changes. Zero data migration. Zero new collections for the three existing balance types.

6. **New balance types use `money_accounts` from day one.** Family wallet, petty-cash float, and consignment balances have no existing collection; they land in `money_accounts` immediately in Phase A. The `money_accounts` unique index (`account_type, account_key`) is built only after the prod dup/null-key pre-clean (known blocker in memory `prod_data_blockers`).

7. **Family-wallet OTP is E6's concern, not E1's.** E1's `request_pool_redeem` calls `E6.send_pool_redemption_otp(...)` and `confirm_pool_redeem` calls `E6.verify_pool_redemption_otp(...)`. E1 does NOT store `otp_hash` on holds — that state lives in E6's `pool_otp` collection. E1 holds only carry an `otp_id` reference. This is the locked resolution from ENGINES.md (Conflicts §1, Cross-engine contracts OTP row). **E1 Phase A defers the family-wallet OTP verbs until E6's transactional slice (Wave 0b) is live.**

8. **E1 is tier-agnostic.** Refund-tier thresholds (auto <₹500 / admin >₹2,000 / super >₹10,000) are enforced by the caller (orders.py, returns.py) reading from E2. E1 enforces only the hard floor (no spend below zero for guarded types) and atomicity.

9. **ENGINES.md locked: all engines call `AuditRepository.create`, never `append_audit_entry` directly.** E1 follows this.

10. **ENGINES.md locked: the Phase-B "dual-write" sentence in E1.md:175 is DELETED from the build scope.** Standalone Mongo (no replica set, verified by zero `with_transaction`/`start_session` in the codebase and `MONGO_REPLICA_SET` env absent) cannot make cross-collection writes atomic. Phase B is DO-NOT-BUILD until a replica set + `with_transaction` helper exist. Phase A ships instead.
