# Packet E5: Tender-routing + reconciliation engine (Phase 2)

## Current behavior (what the code does today -- file:line)

POS captures multi-tender payments correctly, but the accounting/reconciliation side has two gaps:

- **`backend/api/routers/orders.py:376-391`** -- `PaymentMethod` enum: `CASH, UPI, CARD, BANK_TRANSFER, EMI, CREDIT, GIFT_VOUCHER, LOYALTY`. Each `order.payments[]` row carries `payment_id, method, amount, reference, received_by, received_at, idempotency_key` (+ `emi_details`). This capture is correct and **stays untouched**.
- **`backend/database/repositories/order_repository.py:238-287`** -- `add_payment()` enforces over-tender (non-CREDIT cash cannot exceed `grand_total`, 0.01 paisa tolerance; CREDIT exempt), recomputes `amount_paid`/`balance_due`, sets `payment_status`. Authoritative; untouched.
- **`backend/api/routers/finance.py:2249-2294` (`_cash_sales_for_window`) + `:2410-2494` (`/cash-register/close`)** -- EOD cash close ALREADY reads `order.payments[]` **CASH tenders only**, sums cash expenses, compares counted denominations vs expected (`opening_float + cash_sales - refunds - expenses - deposit`). **Only CASH is reconciled; non-cash modes have no by-mode reconciliation.**
- **Tally JV (`backend/agents/nexus_providers.py` / finance JV builder)** -- emits a bare party debit but the **receipt/bank leg is missing or everything books to a single Cash ledger** -- the "everything books as Cash" gap. A 60% UPI / 40% CARD sale does not hit the right bank ledgers.
- **No tender->ledger mapping exists.** There is no `tender_ledger_map` collection; the ledger names are not configurable per scope.

**Net:** instruments other than cash are not classified to their correct accounting ledger, only cash is reconciled at EOD, and an unknown/blank `method` silently behaves as cash. E5 fixes all three by READING `order.payments[]` and writing derived records -- it never re-captures.

## Intended behavior (full intent)

E5 is a **pure, settings-driven mapper + reconciliation reader** over the existing `order.payments[]`:

1. **Canonicalize + route every tender to its correct Tally ledger.** `canonicalize_tender(method, mode)` normalizes each row; `resolve_ledger(tender, map)` maps it to a ledger name. **An unknown/blank method maps to `UNKNOWN -> Suspense A/c`, NEVER silently to CASH** (the bug). Instruments that are NOT cash-in (GIFT_VOUCHER, LOYALTY, CREDIT, STORE_CREDIT) route to **liability/receivable** ledgers, not a bank ledger.
2. **Reconcile by mode for the IST day.** `split_payments_by_mode(payments)` returns `{tender: {collected, refunded, net, count}}`, paise-exact (round only at the boundary). The by-mode net for an order sums to the order total. This feeds the daily close, the cash-variance thresholds (E2), and the finance dashboard.
3. **A daily reconciliation snapshot is the system-of-record for cash-variance.** One `payment_reconciliations` doc per store per IST day; `OPEN` until locked; **lock is atomic** (one guarded `find_one_and_update` on `status:OPEN`) and **immutable** thereafter (hash-chained audit row, immutable even for SUPERADMIN per core philosophy).
4. **The Tally JV gets its missing tender leg.** The day-voucher builder calls E5 to emit one receipt/bank leg per tender (ledger names from the resolved map) so the voucher BALANCES to zero paise (`assert_voucher_balanced`).
5. **Refunds contra the SAME ledger** as the original capture (negative leg), never a separate reversal ledger.
6. **The ledger map is E2-layered** (store override > entity > global), read via E2's `get_policy`/`_get_settings_collection` pattern; the inheritance source is exposed per row in the GET response.
7. **Forward-compatible:** when payment rows later carry `phase`/`edc_terminal` (advance/on-delivery -- a deliberately deferred owner decision) the mapper keys extend without a schema break.

**Locked cross-checks:** E1 (money-guard, DONE) supplies the balance source rows; E2 (settings) resolves the ledger map; E4 (maker-checker) gates a map change and a locked-period re-lock. POS capture + the existing cash-only EOD path are unchanged.

## Delta to build

**New service `backend/api/services/tender_routing.py` (pure, NO DB):**
- `canonicalize_tender(method: str | None, mode: str | None = None) -> str` -- normalize; unknown/blank -> `"UNKNOWN"` (never `"CASH"`).
- `resolve_ledger(tender: str, tender_map: dict, *, is_refund: bool = False) -> str` -- canonical tender -> ledger name via `tender_map`, falling back to `IMS_DEFAULT_LEDGERS`; refund -> same ledger.
- `split_payments_by_mode(payments: list[dict]) -> dict[str, dict]` -- `{tender: {collected, refunded, net, count}}`; sign-split like `_cash_sales_for_window`; paise-exact.
- `IMS_DEFAULT_LEDGERS`: `CASH->"Cash A/c", UPI->"Bank A/c - UPI", CARD->"Bank A/c - Card EDC", BANK_TRANSFER->"Bank A/c", GIFT_VOUCHER->"Gift Voucher Liability", LOYALTY->"Loyalty Points Liability", EMI->"EMI Finance Receivable", CREDIT->"Sundry Debtors", STORE_CREDIT->"Customer Store Credit Liability", UNKNOWN->"Suspense A/c"`.
- `assert_voucher_balanced(legs) -> None` -- paise-exact balance check (precedent: the `_jv_cgst_sgst_split` balance assert) before any Tally emit.

**New service `backend/api/services/tender_reconciliation.py` (DB):**
- `get_effective_tender_map(db, store_id, entity_id) -> dict` -- E2 layering (store > entity > global) via `_get_settings_collection("tender_ledger_map")`.
- `stamp_payment_ledgers(db, order_doc, tender_map) -> dict` -- denormalize `canonical_tender, ledger, ledger_stamped_at` onto each `order.payments[]` row (ADDITIVE, never rewrites capture).

- `lock_reconciliation(db, snapshot_id, actor) -> dict` -- atomic `find_one_and_update({_id, status:"OPEN"}, {$set:{status:"LOCKED", ...}})`; audit via `AuditRepository.create`.

## Data model

**New collection `tender_ledger_map`** (settings; one doc per scope `GLOBAL`/`ENTITY:<id>`/`STORE:<id>`, sparse `ledgers` overrides). Read via the E2 pattern; defaults are `IMS_DEFAULT_LEDGERS` in code.

**New collection `payment_reconciliations`** (daily by-mode snapshot):
```
{ _id, store_id, window_start, window_end (IST day), by_mode: {<tender>: {collected, refunded, net, count, ledger}},
  total_net, status: "OPEN"|"LOCKED", created_at, locked_at?, locked_by? }
```
Index: unique **partial** index `{store_id:1, window_start:1}` where `status="LOCKED"` -- one locked snapshot per store/day (greenfield collection, safe to index).

**`orders.payments[]`** (EXISTING -- additive only): each row gains optional `canonical_tender, ledger, ledger_stamped_at` written by `stamp_payment_ledgers`. **Capture fields are never rewritten.**

**`cash_register_sessions`** (EXISTING -- additive): gains `by_mode_breakdown`.

**Migration:** none required (resolver derives on the fly when stamps absent). The backfill script is idempotent + optional.

## Acceptance tests (INTENT-LEVEL -- a hollow shell must FAIL)

1. **Instruments stop booking as Cash.** A 60% UPI / 40% CARD order produces a JV whose receipt legs hit `Bank A/c - UPI` and `Bank A/c - Card EDC` and **zero on `Cash A/c`**.
2. **Unknown tender -> Suspense, not Cash.** A legacy row `method=""` maps to `UNKNOWN -> Suspense A/c` and is surfaced as `count>0` under UNKNOWN, NOT folded into CASH.
3. **By-mode sums to order total.** `split_payments_by_mode` net over an order's payments equals the order's paid total, paise-exact.
4. **Refund contras the same ledger.** A refund of a CARD payment produces a negative leg on `Bank A/c - Card EDC` (not a separate reversal ledger).
5. **Non-cash-in to liabilities.** GIFT_VOUCHER/LOYALTY/CREDIT route to `Gift Voucher Liability`/`Loyalty Points Liability`/`Sundry Debtors`, never a bank ledger.
6. **Tally voucher balances.** `assert_voucher_balanced` passes (net zero paise) for a mixed-tender day voucher; a deliberately unbalanced set raises.
7. **E2-layered map.** A store override of `UPI -> "Bank A/c - HDFC UPI"` beats the global default for that store only; the GET response shows the inheritance source.
8. **POS capture unchanged.** `POST /{order_id}/payments` and `add_payment` behavior is byte-identical before/after E5 (regression: existing payment tests pass; no new field required at capture).
9. **Lock is atomic + immutable.** Two concurrent `/lock` on one snapshot: exactly one succeeds (guarded `find_one_and_update` on `status:OPEN`); the locked snapshot cannot be mutated via any endpoint; one hash-chained `audit_logs` row exists per lock.
10. **STORE_CREDIT deferred.** STORE_CREDIT is present in the default map but no `order.payments[]` row uses it today and capture is not wired for it (forward-compat only).
