## Intended behavior (full intent)

When an optometrist marks an eye test COMPLETED the system generates a digital handover card that instantly reaches every sales associate and the store manager on that store's floor. The card carries the patient's Rx summary (fetched live from prescriptions), the doctor's product recommendations (free-text, up to 5 rows: category + brand preference + notes), and an optional one-line clinical summary for the sales associate. Sales staff see the card in two places: the topbar bell rings, and a "From doctor" highlighted entry appears at the top of the POS prescription picker when the patient's customer_id is selected. Any sales associate can acknowledge the card (marking it as seen), and can manually mark it "Served" after the transaction to log that the handover led to a consultation. Conversion credit for that served handover is split 50/50 between the optometrist who sent it and the sales associate who marks it served (DECISIONS §3). The handover expires automatically after 8 hours via the existing Mongo TTL mechanism. No WhatsApp is sent; in-app bell only. Product recommendations are free-text (no catalog validation). The feature is piloted at 1-2 stores initially via a per-store feature flag before chain-wide rollout.

## Backend (endpoints + services + which ENGINE calls)

### `POST /api/v1/clinical/tests/{test_id}/send-to-floor`

**Roles:** `OPTOMETRIST`, `STORE_MANAGER`, `ADMIN`, `SUPERADMIN` (reuse `require_roles(*_CLINICAL_ROLES)`).

**Logic:**
1. Feature-flag guard: read `feature_toggles` for `clinical_handover_enabled` on the test's `store_id`. If False → 403 with message "Clinical handover not enabled for this store."
2. Fetch `eye_test` by `test_id`. If not found → 404. If `status != 'COMPLETED'` → 422 "Test must be COMPLETED before sending to floor."
3. `validate_store_access(test.store_id, current_user)` — IDOR guard (reuse `backend/api/dependencies.py:73`).
4. Fetch prescription via `rx_repo.find_by_eye_test(test_id)`. If not found → 422 "No prescription on this test — complete the refraction first."
5. **Idempotency check:** query `handoffs` for `{handoff_type:'CLINICAL_RX', eye_test_id:test_id, expires_at:{$gt:now}}`. If found → return existing `{handoff_id, already_sent:True}` (no duplicate notifications).
6. Resolve recipient list: query `users` for `{roles:{$in:['SALES_STAFF','SALES_CASHIER','STORE_MANAGER']}, store_ids:test.store_id, is_active:True}`. Build recipient sub-docs with `status:'pending'`.
7. Build handoff doc:
   ```python
   {
     "handoff_id": str(uuid.uuid4()),
     "handoff_type": "CLINICAL_RX",
     "uploader_id": current_user["user_id"],
     "uploader_name": current_user.get("full_name") or current_user.get("username"),
     "optometrist_id": current_user["user_id"],
     "optometrist_name": current_user.get("full_name") or current_user.get("username"),
     "title": f"Rx ready — {patient_name}",
     "description": body.clinical_summary,
     "prescription_id": rx["prescription_id"],
     "eye_test_id": test_id,
     "customer_id": test.get("customer_id"),
     "patient_id": test.get("patient_id"),
     "product_recommendations": body.product_recommendations,    # list, max 5
     "clinical_summary": body.clinical_summary,
     "recipients": recipients,
     "created_at": now,
     "expires_at": now + timedelta(hours=8),    # LOCKED 8h TTL
     "validity_days": None,                      # not applicable for CLINICAL_RX
     "mark_served": False,
     "acknowledged_by": None,
     "acknowledged_at": None,
     "served_by": None,
     "served_at": None,
     "parent_handoff_id": None,
   }
   ```
8. `handoff_repo.create(doc)`.
9. For each recipient: `notifications_coll.insert_one({notification_type:'clinical_handover', user_id:recipient_id, title:'New Rx ready — {patient_name}', message:'Dr. {opto_name} has sent an Rx for {patient_name}. Open the handovers section to view.', entity_type:'handoff', entity_id:handoff_id, action_url:'/handoffs/clinical', channels:['IN_APP'], priority:'NORMAL', status:'SENT', created_at:now})`. Fail-soft: any per-recipient failure is logged, never raises.
10. `AuditRepository.create({action:'CLINICAL_HANDOVER_SENT', entity_type:'eye_test', entity_id:test_id, store_id:test.store_id, user_id:current_user['user_id'], detail:{handoff_id, prescription_id, recipient_count:len(recipients)}})`.
11. Return `{handoff_id, recipient_count, already_sent:False}`.

**No E1 / E2 / E4 calls needed** — no money, no approval, no configurable threshold (TTL is locked at 8h by DECISIONS).

### `PATCH /api/v1/handoffs/{handoff_id}/acknowledge`

**Roles:** `SALES_STAFF`, `SALES_CASHIER`, `STORE_MANAGER`.

**Logic:**
- Fetch handoff. Validate `handoff_type == 'CLINICAL_RX'`. Validate caller is a recipient.
- If `acknowledged_by` already set → return 200 idempotently (already acknowledged).
- Single `find_one_and_update({handoff_id, handoff_type:'CLINICAL_RX', acknowledged_by:{$exists:False}}, {$set:{acknowledged_by:uid, acknowledged_at:now}})`.
- Also update the caller's recipient sub-doc: `status:'responded'`, `response:'accepted'` via `handoff_repo.update_recipient(handoff_id, uid, {...})`.
- Audit: `AuditRepository.create({action:'CLINICAL_HANDOVER_ACKNOWLEDGED', ...})`. Fail-soft.
- Return `{ok:True, handoff_id}`.

### `PATCH /api/v1/handoffs/{handoff_id}/mark-served`

**Roles:** `SALES_STAFF`, `SALES_CASHIER`, `STORE_MANAGER`.

**Logic:**
- Fetch handoff. Validate `handoff_type == 'CLINICAL_RX'`. Validate caller is a recipient.
- If `mark_served == True` → 409 "Already marked served."
- Single `find_one_and_update({handoff_id, mark_served:{$ne:True}}, {$set:{mark_served:True, served_by:uid, served_at:now}})`.
- Audit: `AuditRepository.create({action:'CLINICAL_HANDOVER_SERVED', entity_type:'handoff', entity_id:handoff_id, store_id:..., detail:{optometrist_id:handoff.optometrist_id, served_by:uid, conversion_credit:'50_50_split'}})`.
  - The 50/50 split between `optometrist_id` and `served_by` is recorded in the audit `detail` for the SC engine to consume later (DECISIONS §3 — "handover conversion credit split 50/50"). SC is not built yet; the audit row is the source of truth so no credit is lost.
- Return `{ok:True, served_by:uid, served_at:now}`.

### `GET /api/v1/handoffs/clinical-inbox`

**Roles:** `SALES_STAFF`, `SALES_CASHIER`, `STORE_MANAGER`, `AREA_MANAGER`, `ADMIN`, `SUPERADMIN`.

**Logic:**
- Resolve `store_id` from `current_user.active_store_id` (or query param for multi-store roles).

- Return `{handoffs:[...enriched], total}`.
