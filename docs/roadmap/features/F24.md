## Intended behavior (full intent)

A dedicated **Conversion** tab on the Clinical page that answers: "Of all the patients an optometrist examined in a period, what percentage placed a retail order within 7 days?"

**Per-optometrist row (all clinical managers + optometrists themselves):**
- Tests completed in period
- How many of those patients converted (placed a CONFIRMED/PROCESSING/DELIVERED order within 7 days of test completion)
- Conversion rate % (converted / completed * 100)
- Avg days from test to order (clinical efficiency signal)
- Workshop jobs booked (secondary signal: test led to a workshop job)
- Unattributed test count (tests with no `customer_id` that cannot be matched)

**Revenue is ROLE-GATED (DECISIONS §3 - locked):**
- OPTOMETRIST: sees only their own row; columns = tests + converted count + conversion rate %. Revenue column is absent/hidden.
- STORE_MANAGER / AREA_MANAGER / ADMIN / SUPERADMIN: sees all optometrists at their scope + the revenue column (sum of `grand_total` of converted orders) + avg order value.

**Store-level summary bar** (STORE_MANAGER and above): total tests, total conversions, overall rate, total revenue attributed.

**Conversion window:** 7 days (DECISIONS owner answer = option a). A test counts as "converted" if and only if an order from the same `customer_id` with `status NOT IN [CANCELLED, DRAFT]` was created within 7 days of `eye_tests.completed_at`. If a customer has multiple tests before an order, attribution goes to the most recent completed test before the order date.

**Optional month-end snapshot** via TASKMASTER hook (owner answered rolling date-range-only per packet default - rolling date-range is the build target; snapshot is Phase-2 extension).

```python
def get_conversion_dashboard(
    db,
    *,
    store_id: str,
    from_date: date,
    to_date: date,
    conversion_window_days: int = 7,
    include_revenue: bool = False,   # False for OPTOMETRIST callers
    optometrist_id_filter: str | None = None,  # non-None for OPTOMETRIST (self-only)
) -> dict:
    """
    Returns:
      {
        "store_summary": { "tests_completed", "converted", "conversion_rate_pct",
                           "revenue_attributed" (None when include_revenue=False),
                           "unattributed_tests" },
        "rows": [
          { "optometrist_id", "optometrist_name",
            "tests_completed", "converted_count", "conversion_rate_pct",
            "avg_days_to_order",  (float, None when no conversions)
            "workshop_booked",
            "revenue_attributed",  (None when include_revenue=False)
            "avg_order_value",     (None when include_revenue=False)
            "unattributed_tests"
          }, ...
        ]
      }
    Fail-soft: db=None -> {"store_summary": {}, "rows": []}
    """
```

**Aggregation logic (Python-side, not a Mongo pipeline, for readability and Railway Mongo compatibility):**

1. Pull COMPLETED `eye_tests` for `store_id` + `[from_date, to_date]` (using `test_date` string range, same as existing `get_store_tests_in_range`).
2. If `optometrist_id_filter` is set, filter to that optometrist only.
3. Separate tests with `customer_id` (attributable) from those without (unattributed -- logged per optometrist).
4. Collect all distinct `customer_id` values.
5. Pull all `orders` for those customer IDs with `status NOT IN [CANCELLED, DRAFT]` created within `from_date` minus `conversion_window_days` to `to_date` (wide window to catch orders placed after the period's tests). Use `_orders_in_window` pattern with `customer_id: {$in: [...]}}` + datetime bounds.
6. Group by `customer_id` -> list of order `(created_at, grand_total)`.
7. For each optometrist, for each completed test with a `customer_id`, walk that customer's orders to find the earliest order within `conversion_window_days` of `completed_at`. Most recent test before that order = attributed test.
8. Aggregate counts, revenue, avg_days.

**Why Python-side and not a Mongo aggregation pipeline:** The `$lookup` with an `$expr` date subtraction and "most recent test before order" attribution logic requires either multiple pipeline stages with `$map`/`$reduce` (hard to maintain) or a `$lookup` pipeline stage that Railway's Mongo might not efficiently index. Given the data volume (hundreds of tests per store per month, not millions), Python-side is readable, testable, and correct. This can be refactored to a pipeline in Phase 2 if latency becomes an issue.
