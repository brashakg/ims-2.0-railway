## Intended behavior (full intent)

F17 delivers true petty-cash float controls for every store:

**Float account per store.** Each store has an authorised petty-cash float (e.g. ₹5,000). The float is opened by the Store Manager at the start of a period via a credit to the store's `PETTY_CASH` account (E1 `credit`, `account_key=store_id`). Every approved petty-cash payout debits the float atomically (E1 `debit`, single `find_one_and_update` — no dual-write across collections). Balance can never go below ₹0 (floor enforced by E1).

**Receipt required above ₹200.** Any petty-cash expense claim above ₹200 must have a bill uploaded before an approver can approve it. Below ₹200 receipt is optional. Attempting to approve a receipt-less claim above ₹200 returns 400 with a clear message.

**Over-threshold claims go through E4.** The owner-data note (DECISIONS §6, table row "#17 Petty cash (Phase 1-2)") leaves the float amount as "owner data pending", but approval routing is deterministic now: claims above the auto-approval threshold (default ₹500, E2-configurable) require a PIN-gated E4 approval request (`action_type="petty_cash"`). Below threshold, a manager with a set PIN approves inline.

**Low-balance alert.** When the float drops below a configurable threshold (default ₹500 globally, per-store via E2 settings hierarchy), an event `cash.float.low` is emitted on the existing event bus. TASKMASTER and SENTINEL subscribe to alert the store manager in-app. This is a check in the debit path, not a separate cron.

**Status enum aligned.** The runtime enum `PENDING / APPROVED / SENT_TO_ACCOUNTANT / ENTERED / REJECTED` becomes the single truth — `EXPENSE_SCHEMA` is corrected to match. The PETTY_CASH category remains a valid category value in the schema; petty-cash is identified by `category = "PETTY_CASH"` (not a separate collection).

**Float visibility.** A new `GET /api/v1/expenses/petty-cash/balance?store_id=X` endpoint returns the current float balance, low-balance flag, and a ledger of recent movements. Store managers and accountants can see this.

**Audit.** Every float credit and every payout debit emits a hash-chained `audit_logs` row via `AuditRepository.create`. The immutable audit trail is the source of truth for float reconciliation.
