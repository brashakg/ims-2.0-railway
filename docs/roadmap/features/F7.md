## Intended behavior (full intent)

ORACLE computes **burn-rate** (7-day + 30-day rolling velocity) for every active SKU across every store. When a SKU's `days_of_stock_remaining` (on_hand / burn_rate_7d) falls below the **14-day horizon** (DECISIONS locked), ORACLE enqueues one `draft_po` proposal per store, carrying the full reasoning signal in the payload. The payload includes: `burn_rate_7d`, `burn_rate_30d`, `days_remaining`, `projected_stockout_date`, `recommended_qty`, `store_id`, `preferred_vendor_id` (or null with `vendor_missing=True` flag).

Cards for SKUs with no `preferred_vendor_id` are visually flagged with a "No vendor assigned ‚Äî assign before ordering" warning and the Approve button is disabled on those cards.

The existing `GET /reports/purchase/recommendations` endpoint is already the correct aggregation engine; the ORACLE tick calls the same aggregation logic internally (not the endpoint ‚Äî it runs the same Mongo pipeline directly) to keep the agent tick synchronous and avoid HTTP round-trips. The endpoint remains the UI's read path for the dashboard surface.

Separate proposals are generated per store (one `draft_po` proposal per SKU per store per day). The existing `_exec_draft_po` executor stamps `delivery_store_id` from `payload.store_id` so the DRAFT PO is store-attributed.
