## Intended behavior (full intent)

Avinash's accountant (ACCOUNTANT role) can draft a balanced double-entry journal voucher inside IMS — for depreciation charges, rent accruals, bank charges, prior-period corrections — then route it to a senior (ADMIN/SUPERADMIN) for PIN-verified approval before it posts to the ledger. Once POSTED, the entry flows into the P&L read and is exported in the nightly Tally JV. The system cannot be gamed: the drafter cannot self-approve; a closed period blocks posting; every status hop produces an immutable audit row; a POSTED JE can only be undone via a formal reversal JE (not deletion). A minimal seeded chart of accounts (COA) controls which accounts accept manual entries; system-managed balances (GST Output, stock) are protected by `allow_manual_je=False`.

The full lifecycle:

1. ACCOUNTANT or ADMIN drafts a JE: entry_date, description, 2+ balanced lines (debit=credit to the paisa). Status: `DRAFT`.
2. Maker submits (DRAFT -> `SUBMITTED`): in-app bell fires to eligible checkers (ADMIN/SUPERADMIN). Maker cannot be checker — locked in the handler.
3. Checker (ADMIN/SUPERADMIN, not the maker) approves via E4 `approve()` with their PIN (SUBMITTED -> `APPROVED`). Or rejects to `REJECTED` with a mandatory note; maker receives in-app bell.
4. The same checker (or SUPERADMIN) posts (APPROVED -> `POSTED`): `check_period_locked()` is called again (double gate). `posted_at` is stamped. P&L now includes this JE.
5. Reversal: ADMIN/SUPERADMIN only; only on POSTED JEs; auto-creates a mirror JE (debits/credits swapped) on today's date (today's period must be open); both JEs linked via `reversal_of`/`reversed_by`.
6. Nightly Tally export (NEXUS 23:00): POSTED JEs in scope are included as `<JOURNALVOUCHER>` XML alongside sales vouchers.

The ACCOUNTANT owner-decision (DECISIONS.md §6): "Which expense heads you post manually" — answered by seeding COA with a minimal set (Depreciation, Bank Charges, Prior-Period Adjustments, Miscellaneous Expenses) and marking them `allow_manual_je=True`. System accounts have `allow_manual_je=False`. Owner can expand later via SUPERADMIN COA endpoint.

**P&L integration** — `get_pnl()` (finance.py:628) gets a fourth aggregation (additive, no existing queries changed):

```python
# Fourth source: POSTED manual JEs in date range
je_match = {"status": "POSTED", "entry_date": {"$gte": from_dt, "$lte": to_dt}}
if store_id: je_match["store_id"] = store_id
je_pipeline = [
    {"$match": je_match},
    {"$unwind": "$lines"},
    {"$lookup": {"from": "chart_of_accounts", "localField": "lines.account_code",
                 "foreignField": "account_code", "as": "acct"}},
    {"$unwind": "$acct"},
    {"$group": {"_id": "$acct.account_type",
                "net": {"$sum": {"$subtract": ["$lines.credit", "$lines.debit"]}}}}
]
je_by_type = {r["_id"]: r["net"] for r in db.journal_entries.aggregate(je_pipeline)}
# REVENUE accounts: credit increases revenue; EXPENSE accounts: debit increases cost
pnl["je_revenue_adjustment"] = je_by_type.get("REVENUE", 0) / 100  # paisa -> rupees
pnl["je_expense_adjustment"] = je_by_type.get("EXPENSE", 0) / 100
```

4. **Settings hierarchy (E2 seam) — `get_policy()` signature, not `resolve_setting()`.** E4's `_DEFAULT_TIERS` constant is the in-code fallback; when E2 lands, the call becomes `get_policy("approval.refund_tiers", scope={"store_id":...})`. The old `resolve_setting` name in E4.md is dead — never used in this implementation.

5. **IST timezone, FY Apr–Mar, paisa-exact integers.** `entry_date` stored as UTC datetime (from `now_ist()`); paisa-exact as integers (rupees * 100) for `lines.debit/credit`, `total_debit/credit`; FY counter key uses `fy_start_year(now_ist())`.

6. **Status enum string (never colour).** `DRAFT/SUBMITTED/APPROVED/REJECTED/POSTED/REVERSED` are explicit string values stored in the document. The frontend maps these to Tailwind colour classes; the colour is in the UI layer, never the data model. (DECISIONS §Colour-flag-migration.)

7. **No emojis in Python.** All `logger.*` calls and print statements use ASCII tags `[JE]`, `[E4]`, `[APPROVALS]`.

8. **CORRECTIONS.md P0-2 (E3 hash-chain):** JE audit writes go to `AuditRepository.create`, NOT through `audit_chain.append_audit_entry` called directly. The facade handles chaining.

9. **E4 maker-checker hard-enforcement.** For `action_type="journal_entry"`, the `ApprovalEngine.approve()` hard-checks `approver_user_id != requested_by` (HTTP 403 `cannot_approve_own`). This is the LOCKED E4 maker-checker behavior.

## Acceptance tests (INTENT-LEVEL; a hollow shell must FAIL)

Tests in `backend/tests/test_journal_entries.py` and `backend/tests/test_e4_approval_engine.py`. A stub that creates the endpoint but skips validation WILL fail these tests.

1. **Balance invariant (hardlock).** POST `/journal-entries` with `sum(debit) != sum(credit)` by even 1 paisa returns HTTP 422. A perfectly balanced 2-line JE (debit 500000 paisa = credit 500000 paisa) is accepted and lands in DB with `status="DRAFT"`.

2. **allow_manual_je gate.** A JE line referencing account code `2101` (GST Output Tax, `allow_manual_je=False`) returns HTTP 422 with `"account_not_allowed_for_manual_je"`. A line referencing `5001` (Depreciation, `allow_manual_je=True`) is accepted.

3. **Maker cannot approve own JE (hardlock).** User A (ADMIN) drafts and submits a JE. User A calls `POST /journal-entries/{je_id}/approve` with their own correct PIN. Response is HTTP 403 `"cannot_approve_own"`. User B (ADMIN, different user_id) with a valid PIN can approve the same JE — response is HTTP 200, JE status becomes `APPROVED`.

4. **Period lock double gate.** A JE with `entry_date` in a locked period is rejected at DRAFT creation (HTTP 423). A JE created in an open period, but whose period is then locked before POST, is also rejected at the `/post` step (HTTP 423 from `check_period_locked()`).

5. **POSTED JE appears in P&L.** Before posting: `GET /finance/pnl` for the JE date range does NOT include the JE amounts. After `POST /journal-entries/{id}/post`: the same P&L call includes `je_expense_adjustment > 0` for an EXPENSE-type account JE. A hollow shell that just sets `status=POSTED` without touching the P&L aggregation fails this test.

6. **Immutability after POSTED.** Attempt to call `/submit` on a POSTED JE returns HTTP 409. Attempt to call `/post` on a POSTED JE returns HTTP 409. The only valid mutation is `/reverse`, which creates a new JE (does not modify the POSTED one).

7. **Reversal.** Reverse a POSTED JE: response includes a new `reversal_je_id`; the original JE has `status=REVERSED`, `reversed_by={reversal_je_id}`; the new reversal JE has `status=POSTED`, `reversal_of={original_je_id}`, debits/credits are swapped, and it appears in P&L with the opposite effect (net P&L impact = zero after original + reversal).

8. **E4 PIN required.** An ADMIN with no `approval_pin_hash` calling `/approve` gets HTTP 423 `"pin_not_set"`. After `PUT /users/{id}/approval-pin` with a PIN, the same call with correct PIN succeeds. A wrong PIN gets HTTP 403 `"invalid_pin"`. After 5 consecutive wrong PINs within 15 minutes, the account is throttled: HTTP 423 `"pin_locked"`.

9. **E4 single-use consume.** Two concurrent calls to `POST /approvals/requests/{id}/consume` on one APPROVED approval request: exactly one returns ok, the other returns HTTP 409 `"already_consumed"`. The JE can be posted exactly once.

10. **ENABLE_MANUAL_JE flag.** With `ENABLE_MANUAL_JE` unset (or `!=1`), `POST /journal-entries` returns HTTP 503 `"manual_je_not_enabled"`. GET endpoints return normally. With the flag set to `1`, POST is permitted.

11. **Tally JV export.** `GET /tally/journal-jv` for a period with at least one POSTED JE returns valid XML containing a `<JOURNALVOUCHER>` node with correct VCHTYPE, DATE, and ledger entries (debit side = `ISDEEMEDPOSITIVE=No`, credit side = `ISDEEMEDPOSITIVE=Yes`). A DRAFT or SUBMITTED JE does NOT appear in the export.

12. **RBAC coverage-lock.** `test_rbac_policy.py` (the existing coverage-lock test) passes — meaning all 10+ new `/api/v1/finance/journal-entries/*` and `/api/v1/approvals/*` routes have POLICY rows. A build that adds the routes but skips the POLICY rows will fail this test.

13. **In-app bell on submit.** After `POST /journal-entries/{id}/submit`, the `notifications` collection contains at least one document with `recipient_user_id` = an eligible ADMIN/SUPERADMIN and `message` containing the JE number. After `/reject`, the maker receives a bell notification containing the `checker_note`.
