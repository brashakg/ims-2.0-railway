Background Daemon Architecture

Every agent runs continuously as an asyncio background task spawned on FastAPI startup. APScheduler manages each agent's schedule — interval-based, cron-based, or event-triggered. When you toggle an agent OFF, its scheduler jobs are paused (not deleted), so turning it back ON resumes from where it left off.

# FastAPI lifespan — spawns all agents as background daemons from apscheduler.schedulers.asyncio import AsyncIOScheduler from apscheduler.triggers.interval import IntervalTrigger from apscheduler.triggers.cron import CronTrigger scheduler = AsyncIOScheduler() @asynccontextmanager async def lifespan(app: FastAPI): # Load agent configs from MongoDB configs = await db.agent_config.find().to_list() for config in configs: agent = AGENT_REGISTRY[config["agent_id"]]["instance"] if config["enabled"]: scheduler.add_job( agent.background_tick, trigger=config["trigger"], # IntervalTrigger or CronTrigger id=config["agent_id"], replace_existing=True, misfire_grace_time=60 ) scheduler.start() yield scheduler.shutdown()

Per-Agent ON/OFF Toggle

Each agent has an enabled: bool flag in MongoDB's agent_config collection. Flipping this flag pauses/resumes that agent's background jobs instantly — no restart needed.

# MongoDB: agent_config collection { "agent_id": "sentinel", "agent_name": "SENTINEL", "enabled": true, # ON/OFF toggle "schedule_type": "interval", # "interval" | "cron" | "event" "schedule_value": "60s", # how often it runs "last_run": "2026-04-10T14:32:00Z", "last_status": "success", # "success" | "error" | "skipped" "last_error": null, "run_count": 847, "error_count": 3, "avg_run_time_ms": 1250, "config_overrides": {}, # agent-specific settings "toggled_by": "superadmin", "toggled_at": "2026-04-10T09:00:00Z" } # Toggle API — instantly pauses/resumes background jobs PATCH /jarvis/agents/{agent_id}/toggle # Body: { "enabled": true/false } async def toggle_agent(agent_id: str, enabled: bool): await db.agent_config.update_one( {"agent_id": agent_id}, {"$set": {"enabled": enabled, "toggled_at": utcnow(), "toggled_by": current_user.username}} ) if enabled: scheduler.resume_job(agent_id) else: scheduler.pause_job(agent_id)

Key Design Decision

JARVIS is not just another agent — it's the foundation layer. Think of it as the operating system kernel: it provides shared services (DB, Redis, Claude, event bus) that all other agents consume. It cannot be toggled off because without it, no other agent can function. The existing 1,772 lines of code remain mostly intact; the new agents extend this foundation rather than replacing it.

CORTEX — The Orchestrator

Hero Identity: Professor X / Charles Xavier (Marvel)
The world's most powerful telepath doesn't fight — he coordinates. Like Xavier using Cerebro to locate and guide every mutant, CORTEX reads intent and dispatches the right agent for every query.

Why It Exists

Currently, Jarvis uses a flat process_query_async() that classifies intent via keyword matching and then dumps ALL 7 analytics categories into Claude regardless of the query. CORTEX replaces this with intelligent routing — only calling the agents that matter for the query, in the right order, with proper error handling.

Background Tick Implementation

class SentinelAgent(JarvisAgent): agent_id = "sentinel" async def _do_background_work(self): """Runs every 60s when enabled.""" results = await asyncio.gather( self.check_api_health(), # Ping all 31 routers self.check_db_health(), # MongoDB connection pool + slow queries self.check_agent_health(), # Poll all agent health_check() self.check_security(), # Failed logins, token abuse ) score = self.compute_health_score(results) await self.store_health_check(results, score) if score < 70: await self.emit_event("system.degraded", {"score": score})

Key Design: Degradation Scoring

SENTINEL computes a system health score (0-100) by weighting each domain: API (25%), Database (25%), Frontend (15%), Deployments (10%), Data Integrity (10%), Security (5%), Agent Health (10%). A score below 70 triggers automatic notification to CORTEX, which can pause non-critical agent operations to reduce system load.

Key Design: DND & Compliance

India's TRAI regulations require DND (Do Not Disturb) compliance. MEGAPHONE checks the customer's opt-in status before every send. Messages are only sent between 9 AM and 9 PM IST. Bulk campaigns require CORTEX confirmation gate. All sends are logged in notification_log collection with delivery status tracking.

Sync Architecture

class NexusAgent(JarvisAgent): agent_id = "nexus" async def _do_background_work(self): """Hourly sync cycle for all active integrations.""" active = await self.get_active_integrations() for integration in active: try: await self.sync_cycle(integration) except Exception as e: await self.emit_event("sync.failed", { "integration": integration, "error": str(e) }) async def sync_cycle(self, integration: str): 1. fetch_remote_changes() # Pull new data from external API 2. diff_with_local() # Compare against MongoDB state 3. detect_conflicts() # Same entity modified both sides 4. resolve_conflicts() # IMS is master; log overrides 5. apply_changes() # Update local DB + push to remote 6. log_sync_result() # Success/failure + item counts 7. emit_events() # Notify other agents of changes

Key Design: IMS-as-Master

The fundamental design principle is: IMS 2.0 is always the master system. External platforms (Shopify, Razorpay) are consumers of IMS data, not the other way around. When conflicts occur, IMS wins. This prevents the chaos of distributed truth that comes with bidirectional sync. The only exception is online orders from Shopify — those originate externally and are imported into IMS as the source of truth for fulfillment.
